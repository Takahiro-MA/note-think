# note スタイルガイド v2（文体・構成の正典）

> **v2について**: 旧版（過去記事10本分析・旧ペルソナ）は `archive/style_guide_v1.md`。
> 本書は現行の人物像（`ideas/_self-inventory.md`）と「締まったカジュアル」文体に整合させた2026-07改訂版。
> **人物像・声・テーマについて本書と `ideas/_self-inventory.md` が矛盾する場合は、`_self-inventory.md` が勝つ。**

## 1. ペルソナと声（正典は `ideas/_self-inventory.md`）

- 書き手: 「たかみや」(ta_ka_mi_ya)。実像 =「効率を突き詰めて、効率を疑う」47歳の実験者
- テーマは `ideas/_netacho.md` の**4柱**（断酒／身体と暮らしの実験／会社員の実存／AIとの協働）＋雑多コラム。**曜日ローテーションは廃止済み**
- 声の特徴: 問いから入る／データと体感を往復する／メタ認知・自己批判が強い／実務的なのに急に実存的／ユーモアは控えめ・乾いたツッコミ型
- 素材は `_self-inventory.md` にある一次体験のみ使う。**体験・数値の捏造禁止**

## 2. 文体＝「締まったカジュアル」（本節が文体の正典）

- **常体（だ・である）で統一。です・ます調にしない**
- 距離は近いが、馴れ合わない・幼くしない
  - ❌ 避ける: 「だよね」の多用・「笑」・「いや、本当に」「めっちゃ」等のべたつき・過剰な感嘆
  - ✅ 残す: 等身大の本音・控えめな皮肉・断定の切れ・問い→検証・具体
- **短文主体**。一文は長くても60字程度。体言止めを効果的に使う（「36g。軽い。」）
- 箇条書きは必要な時だけ。本文の大半は段落で構成し、一文一文が独立して読める密度を保つ
- 避ける表現: 「〜ではないでしょうか」／過度な敬語／「いかがでしたか？」系の定型締め／カタカナ語の乱用
- 実用ノウハウ系（特に有料）は register をさらに締める（見出し・手順・断定で"元が取れる感"。砕けすぎ厳禁）

## 2.5 エッジの規律 —「ふーん」「当たり前だね」で終わらせない（2026-07追加）

**背景**: 1ヶ月連続投稿の振り返りで「主張が安全圏にあり、読後が『ふーん』で終わる」と自己診断
（`notes/2026-07-07-monthly-review.md`）。スキ率は高いが、記憶に残らず・引用されず・議論を生まない。
以下を執筆時の必須規律とする。

1. **1行要約テスト（最重要）**: 記事の主張を1行に要約する。それが
   「誰もがすでに同意していること」（例:「休息は大事」「自分に合った方法を選ぼう」）なら、**書き直し**。
   良い主張は**読者の少なくとも2割が反対しうる**（例:「サブスクは契約した瞬間に解約しろ」は合格。
   「サブスクは見直そう」は不合格）。
2. **敵を1つ名指す**: 記事が否定する通説・定番アドバイス・世間の空気を必ず1つ明示する
   （「タスク管理術を増やせという記事は、入口の重さを無視している」）。
   敵がいない記事は主張ではなく感想になる。**攻撃対象は通説・構造・仕組みであって、人・属性ではない**
   （`/check-compliance` の「注目狙いの過激化禁止」と両立させる線引き）。
3. **具体の毒を1箇所**: 一次体験の生々しい1シーンを必ず入れる。きれいに要約された体験（「続けるうちに
   変化を感じた」）ではなく、時刻・場所・数字・セリフ・恥や失敗が入った描写
   （「2日目の夜、風呂で急に無常観が来た」）。ここが `_self-inventory.md` の素材の使いどころ。
4. **反転を1回**: タイトルから予想できる結論に直行しない。中盤に「自分もこう信じていた→崩れた」
   「でも実は、問題はそこじゃない」の折り返しを1回入れる。直線の記事は正しくても退屈。
5. **断定で着地する**: 「〜かもしれない」「人それぞれ」「バランスが大事」での締めを禁止。
   締めは自分の選択・行動の表明で終える（読者への一般化はしなくてよい。「僕はこうする」で十分エッジになる）。

※ エッジ＝過激化ではない。声はいつもの「締まったカジュアル」のまま、**主張の位置だけを安全圏の外に置く**。

## 3. タイトル

- 運用手順（候補10本→好奇/共感で採点→本文が回収できるか検証）の正典は `.claude/commands/title.md`
- 型カタログ（v1から継承）:
  1. **「〜した話」型**（体験の結論をタイトルに）
  2. **逆説・意外性型**（常識の逆を突く）
  3. **ダッシュ付き補足型**（メイン──サブ）
  4. **比較・対比型**（AとBの違い）
- 鉄則: 具体的な固有名詞・数字を入れる／読者が「なぜ？」と思う仕掛け／**28〜45字を基本、上限60字**（`/article` の売れる形に合わせる）

## 4. 冒頭（リード）のパターン

**毎回同じ型にしない。以下からローテーションする**（毎回"？"開始にしない）:

1. **結論先出し**: 「結論から言うと、〜」
2. **問題提起**: 「〜という指摘がある。」
3. **シーン描写**: 「視界にAIが常駐する未来に、ずっと興味があった。」
4. **体験の入口**: 「〜という終わりのない競争に疲れていた。」
5. **引用・他者の声**
- 開きの型例（現行の声）: 「〜したいわけじゃない。でも、〜がわからなくなってきた。」「健康的に暮らしているはずなのに、生命力が足りない。」

## 5. 構成と文字数

### 基本構成（1200〜2500字）
```
冒頭: 掴み or 結論（2〜3文）
導入: なぜこの話をするのか（体験のきっかけ）
展開1: 具体的なエピソード・事実
展開2: 構造的な分析・気づき（「なぜそうなるのか」）
主張: 自分なりの結論・スタンス
締め: 余韻を残す一文（or 余談）
```

### 見出し（h2）
- 簡潔に（10〜20文字）。疑問形もOK。数は3〜6個程度

### 記事の種類別テンプレート
- **体験レビュー型**: 掴み → 買った理由 → 最初の印象 → 使い込んで見えたギャップ → 手放した/残した理由 → まとめ
- **考察・論考型**: 問題提起 → 感情的同意 → 「でも構造を見る必要がある」 → 補助線1, 2, 3 → 自分なりの結論
- **気づき共有型**: 結論先出し → Before → 転機 → After → なぜ変わったか → 読者へのメッセージ

### 文字数の目安
| 記事タイプ | 文字数 |
|-----------|--------|
| 考察・論考 | 1500〜2500字 |
| 体験レビュー | 1500〜2000字 |
| 気づき共有 | 1200〜1500字 |
| 比較レビュー | 2000〜4000字 |
| アフィリエイト比較記事 | 2500〜4000字 |

- **有料記事**の構成・文量（無料エリアの黄金順序・有料本編3,000字下限など）は `.claude/commands/article.md` の「売れる形」が正典

## 6. トーンの指針

- **断定はするが、押し付けない**: 「僕はこう思う」ではなく体験と構造で示す
- **他の選択肢を否定しない**: 「合う人には合う」
- **知的な誠実さ**: わからないことは「わからない」と書く
- **尖った主張はする**: 主張にエッジを立てる（穏当すぎる記事にしない）

## 7. アフィリエイト比較記事（v1から継承・現役ルール）

- **`type: free` 固定**・PR表示必須（理由と判定は `/article` 参照）
- 通常記事よりトーンを**具体的・実用的**に寄せる
- 各製品は「良かったところ／引っかかったところ／向いてる人」の3点構成。メリデメは実体験の具体（数値・重量・使用日数）で書く
- スペック表の羅列にしない。「使って初めてわかること」を軸に
- 最終的に「自分はこれを選んだ、理由はこう」と推しを明確に出し、末尾に「こういう人にはこれ」一覧を置く
- Amazonリンク挿入ポイントは `<!-- 🔗 商品リンク: 製品名 -->` のHTMLコメントで明示（リンクはTakahiroが後で手動で貼る）

### 構成テンプレート
```
冒頭: 「結局どれがいいの？」系の問いかけ or 自分の遍歴
各製品セクション（3〜6製品）:
  - 製品名（h2）
  - <!-- 🔗 商品リンク: 製品名 -->
  - 良かったところ（具体的に）／引っかかったところ（正直に）／向いてる人（1行）
結論: 自分が残した1台 + 理由
まとめ表: 「こういう人にはこれ」一覧（※note制約により箇条書きで）
```

### テーマ候補
完全ワイヤレスイヤホン／スマートウォッチ／モバイルバッテリー・充電器／キーボード・マウス／スマホアクセサリ／カメラ・アクションカメラ／スマートホーム／デスク周り・モニター／ランニング・フィットネスギア

## 8. note API制約

- **`<table>`タグは使えない**。noteのエディタが`<table>`をストリップする
- 一覧表が必要な場合は `**太字** → 内容` の箇条書き形式で代替する
- 対応HTMLタグ: `<p>`, `<h2>`, `<h3>`, `<ul>`, `<ol>`, `<li>`, `<strong>`, `<em>`, `<a>`, `<blockquote>`, `<figure>`
- 画像埋め込みは `<figure>` + `embedded-service` 属性（別途API）

## 9. 品質チェックリスト（AI生成時）

- [ ] **常体で統一されているか**（です・ます混入なし）
- [ ] `experience` の素材が `_self-inventory.md` 由来の一次体験か（**捏造なし**）
- [ ] 冒頭パターンが直近3記事と被っていないか
- [ ] 具体的な固有名詞・数字が含まれているか
- [ ] 「体験」が含まれているか（抽象論だけになっていないか）
- [ ] 文が短く切れているか（一文60字以内目安）
- [ ] 見出しが簡潔か
- [ ] **1行要約テスト**を通過しているか（主張が「誰もが同意すること」になっていないか。§2.5）
- [ ] **敵（否定する通説）が1つ名指されているか**（§2.5）
- [ ] **反転が1回あるか**（タイトルから予想できる結論への直行になっていないか。§2.5）
- [ ] 締めが断定か（「かもしれない」「人それぞれ」着地になっていないか。§2.5）
- [ ] 締めに余韻があるか（「いかがでしたか」で終わっていないか）
- [ ] 文字数が範囲内か
