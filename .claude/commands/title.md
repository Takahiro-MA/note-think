note のファネル **「タイトル(CTR) → 開く → 本文(読了) → スキ → おすすめ露出」** の、最初のゲートを最大化する。タイトルは全成果に掛かる最初の係数——ここが弱いと後続が全部目減りする。本文を磨く前に、まずタイトルで「指を止めさせる」。

**対象:** `$ARGUMENTS`（記事ファイルなら core / reader / 本文を読む。テーマ文字列だけでも可）

## 基本方針

- **好奇 ＞ 共感に寄せる**（集客は好奇型が強い、という運用判断）。ただし両タイプを出して比較する。
  - **好奇型**=「お、どういうことだ？」と思わせる（引っかかり・ギャップ）
  - **共感型**=「そうそう、あるよな」と思わせる（あるある・本音の言語化）
- 候補は**最低10本**生成する。多く出してから絞る。

## 最重要の基準：「手段」でなく「読者の便益・悩み」に寄せる（集客の軸）

タイトルは、記事の**具体的な中身（手段・方法）**を説明するより、**読者の悩み・欲・感情**を名指す方が間口が広く、集客に効く。「ドリルではなく"穴"を売る」。

- 例: ×「スマホを"わざと格下げ"したら時間が戻った」（手段寄り・狭い） → ○「スマホ見すぎは、あなたのせいじゃない」（読者の悩み寄り・広い）
- **ただし「広い ≠ ぼんやり」**。悩みは具体的・感情的に名指すこと。抽象的で当たり障りのないタイトルは、広いだけで刺さらない（×「スマホと上手に付き合う方法」）。狙うのは「**抽象的に広い**」ではなく「**特定の痛みを、広い人数が抱えている**」。
- トレードオフ: 便益型はCTRを最大化するが、**本文が約束を回収できることが前提**（回収できないと読了率が落ちて逆に沈む）。集客重視なら便益型、確実性重視なら手段型。実データ（CTR×読了×スキ）が出たら検証する。

## 好奇を作る装置（優先して使う）

- **数字＋禁止/限定**: 「絶対に〜してはいけない3つの〜」「〜する前に決めるべき、たった一つの〜」
- **逆説・裏切り**: 「〜が、いちばん効率の悪いやり方だった」「〜をやめたら、〜になった」
- **欠落/ギャップ**: 「なぜか〜していく」「〜な人だけが知っている」「〜の正体」
- **意外な断定**: 「〜は時間の無駄である」「〜はもう古い」
- **結果先出し / Before-After**: 数字や変化を頭に出す（※その数字が本物のときだけ）
- **ターゲット名指し**: 「〜で消耗している人へ」

## 共感を作る装置

- あるある・小さな違和感の言語化／みんな薄々思ってる本音を先に言う

## ハードルール（釣りの線引き）

1. **タイトルの"約束"は、本文が必ず回収できること。** 回収できない誇張は、開かせても読了率が落ち、note のおすすめで逆に沈む＝自滅。「釣ってOK、ただし回収できる釣りに限る」。
2. **捏造した具体（使っていない/起きていない数字・体験）をタイトルに入れない。** 例:中身が伴わない「コメント14倍」。→ `/check-compliance` とも整合。
3. 景表法・規約の誇大表現（「必ず儲かる」等の断定保証）は不可。

## 採点

各候補を3観点で ◎○△ 評価する:
- **指止め力**（好奇 or 共感の強さ。一覧/検索で手が止まるか）
- **回収可能性**（本文がその約束を果たせるか）
- **具体性**（対象・成果・切り口が入っているか。①公式分析の勝ち筋）

好奇型を優先しつつ、**回収可能性が△のものは落とす**。

## 出力

1. 候補一覧（各行: `[好奇/共感][型] タイトル ── 指止め◎/回収○/具体○ ── 一言:なぜ指が止まるか`）
2. **推奨トップ2〜3**（理由つき）
3. 第一候補を**太字**で1本。

## 確定後の反映

第一候補で進める合意が取れたら:
- 記事ファイルの `# タイトル行` を差し替え（必要なら冒頭リード・core との整合も微調整）
- note 下書きがある場合は `python3 -m note_poster update <id> "<新タイトル>" <body>` で同期するか確認する
- 長さ: note のフィード表示を意識し、引っかかる語を**前半**に置く。説明的な後半は削ってよい
