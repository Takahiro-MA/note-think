# note記事制作ワークフロー — フレームワーク移植ガイド

> **⚠️ 注: 本書は設計の歴史的経緯の記録。現行仕様は `04_integrated_design.md` が正。本書が提案するPhase 6段階・グループA/B/Cのスキル分類は採用されず、04で軽量化されて実装された。**

> このドキュメントは、Kindle書籍執筆プロジェクト（「田中さんシリーズ」）で運用実績のあるワークフローから、**コンテンツ非依存の「枠組み」だけを抽出**し、note記事制作に転用できる形にまとめたものです。
> **文体・文量・文化はnote向けに作り直す前提**で、プロセスの骨格・skill構成・状態管理の考え方だけを移植します。
> このファイル単体で、別プロジェクトの設計土台として使えるように書いています。

---

## 0. 元になっているワークフローの正体

Kindle側のワークフローは、次の3層構造でできています。上の2層はコンテンツに依存しないので、ほぼそのままnoteに持ってこられます。

```
┌─────────────────────────────────────────┐
│ 層1: Phaseゲート（段階管理）              │ ← 転用可
│   仕様→AIレビュー→人間承認→執筆→           │
│   AIレビュー→修正→人間承認→完成            │
├─────────────────────────────────────────┤
│ 層2: skill群（各Phaseを駆動するコマンド）  │ ← 構造は転用、中身は一部入替
│   思考モード / レビュー / 修正適用 など     │
├─────────────────────────────────────────┤
│ 層3: 生成物の定義（仕様書・本文の中身）     │ ← ここだけ作り直し（note固有）
│   文体・章構成・キャラ・文量               │
└─────────────────────────────────────────┘
```

---

## 1. コアとなる3つの思想（これがフレームワークの本体）

1. **仕様書ファースト** — 本文を書く前に「記事設計書」を固める。Kindle側の実感値は「**仕様書の精度が品質の8割を決める**」。note でも、いきなり書き始めず "何を・誰に・どの順で・どう締めるか" を先に1枚にする。

2. **段階ゲート（Phase）** — 一気に完成させない。各段階の終わりに「AIレビュー」と「人間（あなた）の承認」を挟み、前段が固まってから次へ進む。手戻りを後工程に持ち越さない。

3. **多視点・並列レビュー** — 1つのレビューを1人格でやらない。役割の違う複数の視点を**並列に**走らせ、優先度付き（必須/推奨/任意）の指摘リストに統合する。

---

## 2. Phaseフレームワーク（note向けに簡略化した提案）

Kindle版は10段階（Phase 0→1→1R→1A→2→3→4→5→6→Published）。noteは1記事が軽いので、**6段階に圧縮**するのを推奨します。人間ゲート（あなたの承認）を2箇所に置くのがポイントです。

| Phase | 名前 | 中身 | ゲート |
|-------|------|------|--------|
| **0** | ネタ検証 | そのネタで1本書く価値があるか（GO / NO-GO / 寝かせる） | — |
| **1** | 記事設計書 | コアメッセージ・読者・構成・締めを1枚に（§6のテンプレ） | — |
| **1R** | 設計のAIレビュー | 設計書を多視点レビュー→修正反映 | **人間承認①**（設計OKか） |
| **2** | 初稿執筆 | 設計書に沿って一気に書く | — |
| **3** | 原稿レビュー＆修正 | 原稿を多視点レビュー→優先度順に修正 | — |
| **4** | 仕上げ・公開準備 | タイトル/見出し最終化・冒頭フック・CTA・タグ | **人間承認②**（公開GOか） |

> 「設計の人間承認①」が効きます。設計が固まる前に本文を書くと、note でも「結局何が言いたいのか分からない記事」になりがち。Kindle側でも、この著者レビューゲート（Phase 1A）がボトルネックでもあり品質の要でもありました。

## 3. 状態管理ファイル（全記事を1枚で管理）

Kindle側の `book-status.md` に相当するものを `article-status.md` として持つと、複数記事を並行して回せます。テンプレ：

```markdown
# 記事ステータス一覧

Last Updated: YYYY-MM-DD

## ステータス凡例
| Phase | 意味 |
|-------|------|
| Phase 0 | ネタ検証（GO/NO-GO/寝かせる） |
| Phase 1 | 記事設計書作成 |
| Phase 1R | 設計AIレビュー＆修正（人間承認①） |
| Phase 2 | 初稿執筆 |
| Phase 3 | 原稿レビュー＆修正 |
| Phase 4 | 仕上げ・公開準備（人間承認②） |
| Published | 公開済み |

## 記事一覧
| # | タイトル仮 | 設計書 | Phase | 通過済み | 公開先/URL | 備考 |
|---|-----------|--------|-------|---------|-----------|------|
| 1 | （ネタ）  | `articles/xxx/design.md` | Phase 1R | 0→1→**1R** | — | — |
```

## 4. skill構成（転用度で3グループ）

Claude Code のスラッシュコマンド（`.claude/commands/*.md`）として実装します。形式は `--- description: ... ---` のフロントマター＋本文の指示文。

### グループA：そのまま転用（コンテンツ非依存の思考モード）
ほぼ無改造でnoteに持ってこられます。ネタ出し〜設計フェーズで使う。

- `/brainstorm` — アイデア発散モード（ネタを広げる）
- `/sparring` — 議論・壁打ちモード（主張を鍛える）
- `/evaluate` — 評価・比較モード（複数ネタ/構成案を採点）
- `/decide` — 意思決定モード（優先度を決める）
- `/spec` — 仕様書作成モード（→ note では「記事設計書」作成に充てる）
- `/archive` — 議論内容を設計書に保存するモード

### グループB：構造を転用・中身を入替（多視点レビュー）
「3つのSubagentを並列起動して優先度付き指摘を返す」骨格はそのまま、**レビュアーの視点だけnote向けに差し替え**ます。雛形は§5。

- `/review-design` — 記事設計書の多視点レビュー（Kindleの `/review-thinkinghub` 相当）
- `/review-draft` — 原稿の多視点レビュー（Kindleの `/review-book` 相当）
- `/fix-review` — レビュー指摘を優先度順（必須→推奨→任意）に修正適用

### グループC：note固有で新規作成
- `/article-start` — 記事プロジェクト開始（Phase 0→1の入口、設計書の雛形生成）
- `/write-article` — 設計書に基づいて初稿を書く

## 5. レビューskillの雛形（note版・並列多視点）

Kindleの `/review-thinkinghub`（3視点並列Subagent→優先度付き統合）を、note向けに視点を差し替えた雛形です。**そのまま `.claude/commands/review-draft.md` として置けます。**

```markdown
---
description: note原稿の3視点レビューを並列実行
---

指定された原稿ファイルに対して、3つのSubagentを並列起動してレビューを実行します。

**対象ファイル:** $ARGUMENTS

## 共通の参照
- 記事設計書（design.md）— 設計意図とのズレを確認
- 文体・トーンガイド（あれば）

## レビュアー1: 編集者（構成・読ませ方）
- リード文（冒頭2〜3行）で読者を掴めているか。離脱しないか
- 1記事1メッセージになっているか。主張がブレていないか
- 見出しだけ拾い読みして筋が通るか
- 段落の長さ・改行・リズムがnoteとして読みやすいか
- タイトルと中身が一致しているか（釣りになっていないか）
優先度付き（A:必須 / B:推奨 / C:任意）で指摘を返す。

## レビュアー2: ターゲット読者（共感・有益性）
- 想定読者の悩み・関心に刺さっているか。「自分ごと」と感じるか
- 読者が「読んでよかった（学び/共感/行動）」と思える持ち帰りがあるか
- 前提知識を置いてけぼりにしていないか
- 「で、結局どうすれば？」が残らないか
優先度付きで指摘を返す。

## レビュアー3: 拡散・回遊（タイトル・フック・CTA）
- 検索/タイムラインで「読みたい」と思えるタイトルか
- 冒頭フックが弱くないか（最初の数行で続きを読ませるか）
- スキ・シェア・フォローしたくなる要素（引用したい一文、共感ポイント）があるか
- 読後のCTA（次の記事/マガジン/フォロー/コメント誘導）が自然か
優先度付きで指摘を返す。

## 実行方法
3つのレビュアーを独立したSubagent（Task tool）で並列起動し、完了後に
1. A（必須修正）2. B（推奨修正）3. C（任意）の順で統合報告する。
```

> `/review-design`（設計書レビュー）も同じ構造で、視点を「コアメッセージは1つに絞れているか / 読者と悩みが具体的か / 構成に飛躍がないか」等に差し替えればOK。

## 6. 記事設計書テンプレ（thinking-hub軽量版）

Kindleの `thinking-hub.md`（16セクション）は書籍向けに重いので、note は **7項目**に圧縮します。これを `articles/<slug>/design.md` として1記事ごとに作り、Phase 1で埋めます。

```markdown
# 記事設計書: <仮タイトル>

## 1. コアメッセージ（1記事1主張）
この記事で読者に「これだけは持ち帰ってほしい」一文。

## 2. ターゲット読者と悩み
- 誰が読む（具体的な状況）
- その人が抱える悩み・知りたいこと

## 3. タイトル・見出し候補
- タイトル案 3つ（検索/SNSで刺さる言葉）
- 見出し構成（H2レベル）

## 4. 記事構成（リード→本論→まとめ）
- リード（冒頭フック：なぜ今あなたが読むべきか）
- 本論（節ごとの要点を箇条書き）
- まとめ（コアメッセージの再提示）

## 5. キラーフレーズ／引用したくなる一文
SNSでシェアされる・心に残る言い回しを2〜3個。

## 6. CTA（読者に促す行動）
読後にしてほしいこと（フォロー/マガジン購読/コメント/次記事）。

## 7. 文体・トーン・文量
- 文体（です・ます / だ・である / 語りかけ 等）
- トーン（カジュアル/専門的/エモい 等）
- 想定文量（例: 2,000〜4,000字）
```

---

## 7. 移植の最初の一歩（別スレッドでやること）

1. このドキュメントを土台に **`article-status.md`** を作る（§3）
2. グループAの思考モードskillを **`.claude/commands/`** にコピー（Kindle側からそのまま流用可）
3. **`review-draft.md`** を §5の雛形で1つ作る（note視点に調整）
4. **`design.md` テンプレ**（§6）で試しに1本設計し、Phase 1R（設計レビュー）まで回してみる
5. 回してみて、Phase数・レビュー視点・設計項目を**note文化に合わせて削る/足す**
